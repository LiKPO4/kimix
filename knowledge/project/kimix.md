# Kimix

Kimix is an Electron application with a React renderer and a Node-based main process. It provides project-aware conversations, streamed Markdown, approval and question cards, session history, local Skills, MCP visibility, and cross-platform packaging.

# Boundaries

* The renderer lives under `src/` and communicates with privileged capabilities through the preload bridge.
* The Electron main process lives under `electron/` and owns filesystem, settings, session, runtime, and IPC integration.
* The repository-root `start-kimix.bat` is the canonical local launcher for the built app. Its default mode requires complete `out/` output, rebuilds dirty worktrees, and records the current Git HEAD in `out/.kimix-build-fingerprint`; a missing or mismatched fingerprint rebuilds before launch. Startup also stops Kimix Electron, packaged, compatibility-copy, and related build processes so Electron's single-instance lock cannot hand a new launch to an older copy. `--fast` is the explicit bypass for already-verified output, while `--dev` remains the hot-reload path. The no-BOM PowerShell helper stays ASCII-safe because Windows PowerShell 5.1 can decode UTF-8 exception text as ANSI and lose quote terminators.
* Desktop notifications cover completed turns, pending clarification questions, and pending tool approvals under one user-selected visibility policy. Notification bodies remain generic unless the user enables content previews. Every notification carries the Kimix UI session identity; clicking it restores and focuses the window, resolves local, runtime, or official session aliases, switches to the corresponding project when known, and opens that chat. On Windows, notification and single-instance activation use a bounded topmost pulse before `moveTop` and `focus`, then restore the prior always-on-top state, because a background process cannot reliably enter the foreground with `focus` alone. Pending request notifications deduplicate by runtime session plus stable request identity so replayed event snapshots cannot repeatedly alert the user.
* Update history is sourced from the latest three GitHub Releases, never from a bundled historical timeline. The dialog renders each complete Release body as Markdown instead of truncating it to a summary. The main process prefers the GitHub Releases list API and falls back to the repository Releases Atom feed on anonymous API rate-limit responses (403/429); both sources remain GitHub-owned. Atom enclosure links are accepted when present, malformed percent-encoded tags cannot abort parsing, and an asset-less Atom result opens its GitHub Release page instead of offering an impossible in-app package download.
* Kimi Code is the authoritative agent runtime. Kimix presents and adapts its APIs; it does not define a second agent protocol.
* Composer sends the user's text without a local clarification-mode preamble. Official Kimi structured questions, question cards, and notifications remain supported, but Kimix no longer stores or syncs the retired three-state clarification setting. Canonical history mapping still strips both legacy `【Kimix 需求澄清工具：…】` and later `【Kimix 需求澄清：…】` wrappers; local persistence hydration strips the same wrappers from user events and room-message mirrors before rendering, while leaving steer messages untouched. Cache mapping version 4 treats either wrapper in a local user event as explicit repair evidence so old sessions can also accept canonical original text instead of preserving the polluted bubble.
* The chat shell owns the viewport height. Its outer height is fixed directly to the Electron viewport with `height/max-height: 100dvh`; ancestor percentage heights and Tailwind `h-full` are not a definite-size boundary for this Grid. The shell uses `48px minmax(0, 1fr)` tracks and the chat panel uses `toolbar + minmax(0, 1fr) + Composer`; every nested viewport remains shrinkable with `min-height: 0`, and only the inner chat scroll area owns message overflow.
* ContextBar popovers are renderer overlays, not children of the clipped chat grid. Workspace, usage, and model panels render through a `document.body` portal with fixed coordinates derived from their trigger, update on viewport/anchor changes, and clamp to the viewport margin. Git overlays keep one data/action implementation in `LongTaskInspectorPanel`, but AppShell may mount that overlay host with the inspector body hidden. The footer branch control opens the Git graph directly through its own signal, while the inspector's Details action remains the entry point for commit details; neither footer entry point may change right-inspector visibility. On a newly mounted overlay-only host, project-path initialization and request-generation invalidation must run before external open signals are consumed, otherwise the first Git request is immediately marked stale and its loading state never settles. Git graph commit subjects own the primary description line; all branch/tag decorations returned by the bounded data layer render on a separate wrapping metadata line without a fixed badge-width cap or `+N` summary, so refs do not consume the subject's truncation budget and remain individually visible.
* Managed plan usage preserves the official optional `extraUsage` BOOSTER wallet alongside 5-hour and weekly windows. It is read-only account data: Kimix displays monthly spend, the enabled monthly charge limit or an explicit unlimited state, and remaining balance, but does not invent a local enable, top-up, or limit-edit action. SDK-normalized cents are authoritative; the direct `/usages` fallback applies the official fixed-point conversion to raw wallet balances. Invalid, non-BOOSTER, or zero-total wallets remain absent rather than producing an empty card.
* Expanded sidebar projects use a compact three-level rhythm without collapsing rows: 6px between project sections, 4px between a project header and its session list, and 2px between session rows. Project and session row heights remain 36px and 32px so density changes come from removing accumulated empty space rather than shrinking text or controls.
* The chat main grid must declare `grid-template-columns: minmax(0, 1fr)`; an implicit `auto` column can retain a child's intrinsic width even when the main element's border box shrinks for the right inspector. The session toolbar additionally uses `minmax(0, 1fr) auto` title/actions columns and sets `min-width: 0` plus `width/max-width: 100%`. The title column owns truncation while the fixed action column remains inside the main area.
* The resizable right inspector supports a 280px minimum width. Three icon-and-text Git actions cannot fit one row after inspector and card padding at that width, so the Git action grid switches below 340px to two columns with Push spanning the second row; wider inspectors retain three equal `minmax(0, 1fr)` columns.
* Editing a queued prompt is a destructive restore-to-Composer action, not a separate edit mode: the queued item is removed, its text and attachments become the ordinary draft, and subsequent send/queue behavior is identical to newly typed input. Kimix does not retain an edit snapshot or expose Save/Cancel Edit controls.
* Sidebar project rows own expansion only. Clicking a project toggles its session list without changing the active project/session or creating a conversation; only a session row changes the active conversation, and the explicit New Conversation action creates one.
* Image preview keyboard navigation is owned by the shared overlay, not individual thumbnails. Composer attachments, user messages, and steer messages provide their previewable image group; Left/Up selects the previous image and Right/Down the next without wrapping at either boundary, and Escape closes single- or multi-image preview. An open drawing board retains keyboard ownership.
* Historical image recovery follows official Kimi persistence formats instead of requiring an OS drag path. Kimi Web uploads live under the session `uploads/` directory, while Kimi Server prompts persist native `{ type: "image", source: { kind: "base64", media_type, data } }` content. Kimix maps both legacy `image_url` and native `image/source` records back to displayable data URLs. Official history remains authoritative for timeline bodies and process events, but every whole-history replacement must reconcile renderer-local attachment metadata first: pasted image data URLs and OS drag paths are matched by message type, normalized content, and nearest timestamp, then merged into the canonical user event so richer tool history cannot downgrade a fresh preview to a name-only placeholder.
* The user-facing font-size setting keeps the historical 15px chat and Composer baseline, with the sidebar one pixel smaller. The first font-size baseline migration upgrades the accidentally introduced persisted 14px default once, records `fontSizeBaselineVersion: 1`, and never overrides a later explicit 14px user choice.
* Assistant process rendering preserves the official step timeline. Local wire history must retain `tool.call` and `tool.result` nested inside `context.append_loop_event`; thinking parts before and after those calls remain separate phases, a think part sharing the tool timestamp is ordered before that tool, and each collapsed phase uses its final natural paragraph as the summary title. In Kimi Web mode, only the latest renderable Assistant process expands when a session opens or a new turn begins; historical process lists stay unmounted behind their summary row while final Assistant Markdown remains visible, and a user may explicitly reopen any historical process. A completed pre-turn compaction belongs immediately after that turn's user message and before every Assistant status, process, tool, or final-text item, regardless of event arrival order. Its generated summary stays unmounted behind a collapsed disclosure row until the user explicitly expands it. Foldable summaries and non-foldable single-paragraph process items share the same secondary typography; foldability changes interaction only, while expanded details remain muted and final Assistant Markdown remains primary text. Interactive rows inside an expanded process card own the full card width for hover and dividers; their horizontal content inset belongs on each row, not on the list wrapper, so the interaction surface reaches both card edges without shifting text alignment.
* Persisted Kimi history carries a Kimix mapping-version marker. When the mapping changes, the active or selected stale cache is reloaded once from official history and replaced when the canonical timeline contains richer process events; non-empty cached messages alone must not permanently block history repair.
* Chat auto-follow distinguishes browser scroll events from user intent. Content shrink, history replacement, Markdown settlement, and browser scroll clamping may move `scrollTop` upward without user input and must not disable the session-open bottom-settle window. Following mode owns the canonical bottom `max(0, scrollHeight - clientHeight)` and observes both streamed content and the scroll viewport, because Composer, Todo, and queue height changes can shrink the viewport without mutating the message list. A content-version layout commit snapshots the pre-mutation bottom distance and restores it synchronously when the new stream content is shorter. The first final-content render keeps the live thinking viewport geometry until its pre-collapse snapshot is captured, then removes the process detail in one protected transaction instead of exposing a live-to-teaser-to-collapsed double shrink. Only recent wheel, touch, middle-button autoscroll, scrollbar-drag, or navigation-key intent may hand scrolling control to the user. Explicit downward intent that reaches the bottom threshold may resume following; a layout clamp at the bottom without such intent may not.
* Chat scroll diagnostics must stay off the normal streaming hot path. Continuous tool output can emit content and resize updates many times per second; routine `contentVersion`, `ResizeObserver`, and no-op manual-anchor restoration logs create renderer/main-process work that competes with painting and timers. Keep high-frequency paths silent or throttled, and reserve detailed diagnostics for anomalies such as jump-to-top or actual anchor movement.
* Once user intent takes manual control of the chat viewport, layout-only mutations must preserve a rendered message anchor instead of trusting raw scroll offsets alone. Permission controls, footer/menu resize, runtime status snapshots, deferred Markdown settlement, and process-to-final transitions can all submit layout after the user scrolls. Only explicit input advances the manual-scroll generation and invalidates the previous anchor; programmatic restoration and browser clamping must not masquerade as new input, and high-frequency layout updates must not postpone the input-owned idle capture timer. Ordinary commits restore the fresh rendered-message anchor, while a known process collapse samples a surviving DOM element at the reading line and restores it without the generic suppression window. If shrink below the viewport makes the browser's new maximum `scrollTop` too small, `ChatThread` carries the exact deficit as temporary tail compensation; real final-content growth consumes it, and explicit return to the bottom or session switching clears it. Browser-native scroll anchoring remains enabled as the first-line stabilizer. Small ordinary resize anchoring may cap movement to avoid jitter, but an explicit collapse transaction and recovery from a confirmed jump must allow the full required delta.
* Startup and session switching use bottom-first rendering. Persisted restoration must mark the active official session `isLoading` before it enters the session store; a bootstrap-only gate is too late and can expose a stale tail. The startup canonical load is the sole history writer for that active session, while background repair must exclude it. Kimix shows a centered synchronization state until the canonical load completes, then renders the eager tail once. Server snapshot replay propagates each message's official `created_at` to every derived frame and never treats protocol sequence numbers as wall-clock timestamps; SDK wrapped wire messages likewise preserve message-level or outer-record time. The eager tail uses a bounded context window rather than a fixed raw-item slice: trailing unanswered user turns may extend the window backward to retain recent completed Assistant answers, but the window remains capped and never invents absent replies. After the small tail paints, the first visible batch is prepended while the bottom-origin coordinate system remains active, so background growth extends upward without moving the bottom viewport. Initial-tail completion never depends on whether any items were hidden: short conversations can contain a single extremely tall Markdown answer and must still leave reverse coordinates. Before user interaction, Kimix switches to normal scroll coordinates in a layout effect and sets the real bottom in the same frame. Loading history beyond the first batch remains explicit; a rendered message key and viewport offset preserve position instead of a raw `scrollHeight` delta.
* Local wire parsing must preserve every recognizable historical event. Memory and DOM pressure are controlled by ChatThread's bounded initial tail and explicit older-history expansion, never by silently deleting the earliest parsed events; otherwise a long session can reach a false top with no remaining-history affordance.
* Markdown prose never expands the main chat viewport horizontally. Paragraphs, list items, quotes, headings, and inline code use visual-only emergency wrapping for long paths and identifiers; copied source text remains unchanged. Fenced code blocks, tables, and display math own any necessary local horizontal scrolling, while the chat scroll container itself hides horizontal overflow.
* A running Assistant renders Markdown as stable top-level blocks rather than reparsing one ever-growing document. The renderer uses the same `marked` Lexer strategy as the official Web streaming path to preserve block boundaries, keeps completed blocks memoized, and reparses only the changing tail with the existing `ReactMarkdown` components and plugins. Completed messages retain the established static renderer, so the streaming optimization does not replace Kimix's settled-history presentation. The lightweight Lexer dependency is intentional: importing the full Streamdown package would also bundle unrelated Shiki, Mermaid, themes, and language resources.
* Sidebar recency means conversation activity, not generic session metadata mutation. Once timeline bodies are available, the displayed time comes from the latest user, steer, or Assistant message; official catalog `updated_at` is only a fallback for lightweight entries because runtime resume and metadata synchronization may also change it.
* Official session catalog titles prefer a non-default official title over first/last prompt fallbacks, so lazy history hydration cannot visibly rename the row. A transient `isLoading` flag belongs only to the currently loading view; non-current rows show a spinner only for genuine runtime activity.
* Official empty-session visibility is shared by the shell and sidebar. A default-title Kimi session older than the creation grace period with no user or steer content stays hidden even before asynchronous project catalog reconciliation completes; freshly created sessions and sessions with real user content remain visible.
* Kimi placeholder titles are bilingual: both `New Session` and `新会话` are defaults even when legacy or fork metadata marks them custom. Catalog and live metadata paths reject either placeholder and fall back to meaningful prompt text instead of renaming only after history hydration.
* Active project/session persistence is written by store changes and flushed again on the real `beforeunload` boundary. React effect cleanup must never write active context because Strict Mode and HMR run cleanup before startup restoration and can replace a valid saved session with `null`. Startup has one navigation owner: settings hydration may populate the recent-project catalog but cannot select a project, and active-context writes remain gated until persisted sessions and the frozen startup context have resolved. A saved project with no active session remains intentionally empty instead of falling through to another project's latest conversation; only the complete absence of saved context permits that recent-conversation fallback.
* Sidebar expansion is a path-keyed user preference, separate from active project/session navigation. Restored expansion state, including an explicitly empty set, is authoritative on first startup paint; the active project is auto-expanded only when no preference has ever been saved. Later explicit session navigation may expand its project, while manually collapsing the active project remains stable across restart.
* Theme presets imported from the active Kimi Code `themes` directory are a cached filesystem projection. A rescan must reconcile additions, updates, and deletions from that directory while preserving presets owned by other sources. Removing a Kimix record and deleting its source are separate actions; source deletion requires confirmation and a main-process path guard restricted to direct JSON children of the active themes directory.
