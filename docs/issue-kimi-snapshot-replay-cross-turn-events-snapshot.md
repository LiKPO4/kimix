# Kimi Server 快照跨轮串消息事件快照

抓取时间：2026-07-19（Asia/Shanghai）

## 用户侧现象

- 会话在用户发送“你好还在吗”后，将两段语义不同的助手正文显示在同一轮中。
- 同一消息头同时显示“正在输出 1939分48秒”，侧栏保持转圈，但底部连接状态已经是“已连接”。

官方当前助手消息只有一段正文；截图中的第二段正文可精确匹配约 8.5 小时前的历史消息。因此这不是模型或 Server 在一轮内生成两次，而是 Kimix 快照回放的消息归属错误。

## UI 时间线诊断

打包应用未开启 CDP，无法直接导出当时的 React `renderItems`。截图提供的可见投影与 Server 快照足以确定以下差异：

1. Server 当前轮只有消息 `000143`，UI 却把 `000048` 的正文投影到当前轮。
2. Server 已无活动 prompt，UI 仍保留未完成 Assistant 和侧栏 busy 投影。
3. UI 的超长计时与当前消息的官方旧时间一致，说明回放终态没有关闭对应 Assistant。

## 根因

1. `mapKimiCodeEvent` 只读取数值 `time`，忽略 Server 快照的 `created_at`，历史帧退化为回放到达时间。
2. `context.append_loop_event` 解包时没有完整保留快照时间和回放元数据。
3. `turn.ended` 映射丢失 `snapshotMessageId`，`mergeEvents` 又按“最后一个未完成 Assistant”合并或收口，没有先按官方稳定消息 ID 选目标。
4. 快照终态过滤把“近期未完成时间线”当成权威运行态，回放内容会令自身变为 active，进而跳过自己的终态，形成永久未完成循环。

## 修复不变量

- 快照帧必须保留官方时间与稳定消息 ID，嵌套事件也不例外。
- 带稳定 ID 的正文和终态只允许更新同 ID Assistant。
- 迟到且早于最新用户边界的历史消息按官方时间插回历史轮，不得并入当前未完成 Assistant。
- 只有权威 runtime 归属可以阻止当前轮历史终态收口；时间线启发式 active 不能否决权威快照终态。
- 活跃新轮存在时，旧稳定消息的历史终态仍必须关闭旧消息，不能被全局跳过。
- 若旧版本已把稳定消息 ID 挂到错误用户轮，官方快照可凭“同 ID、不同用户归属”的确定证据覆盖更长的污染正文；普通较短快照仍受单调性保护。
