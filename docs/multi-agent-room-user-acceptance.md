# Kimix 多 Agent 房间用户验收清单

验收版本：`v2.15.35`

验收范围：内部开发 gate，不推送、不发布。建议使用干净 Git 工作区或专门测试项目，避免多个 Agent 同时修改重要文件。

## 0. 准备

- [ ] 左下角版本显示 `v2.15.35`。
- [ ] 设置 → 实验功能 → “多 Agent 房间（内部验收）”显示“验收中”。
- [ ] Composer `+` 菜单顶部显示“添加 Agent”，初始计数为 `1/4`。
- [ ] 添加弹层能看到现有 Kimi Code Provider 和模型，不需要重新录入 API Key。

## 1. 两 Agent 精确路由

1. 在普通会话中添加第二个 Agent，例如 `Reviewer / @reviewer`。
2. 先只选择主 Agent 发送一条消息，再只选择 Reviewer 发送另一条消息。
3. 分别使用 `@主Agent`、`@reviewer` 发送消息。

验收：

- [ ] 未被选择或未被 `@` 的 Agent 不创建回复块、不接收消息。
- [ ] 每个回复块显示正确 Agent 和模型。
- [ ] Agent 名称、mention、权限和模型修改只影响目标 Agent。

## 2. 同 Provider 不同模型与跨 Provider

1. 让两个 Agent 使用同一 Provider 的不同模型，向两者发送同一问题。
2. 再把其中一个 Agent 换成另一 Provider 的模型，重复发送。

验收：

- [ ] 两个 Agent 保持独立上下文，不会引用另一个 Agent 未被用户转发的内容。
- [ ] 模型标签与实际选择一致。
- [ ] 一个 Agent 失败、不可用或等待交互时，另一个 Agent 仍可继续运行。

## 3. 四 Agent、长名称与窄窗口

1. 添加到 4 个 Agent，至少一个使用接近 40 字符上限的显示名称。
2. 缩窄窗口，展开 Agent 选择器。

验收：

- [ ] 计数显示 `4/4`，继续添加按钮禁用。
- [ ] 长名称截断但不挤压状态、选择标记和操作按钮。
- [ ] Agent 列表、共享目录风险提示、运行状态之间有清晰间距。
- [ ] 风险提示明确说明所有 Agent 共享同一工作目录。

## 4. 并行、历史展开与滚动

1. 同时向至少两个 Agent 发送任务。
2. 最新 Agent 运行时滚动到较早消息，展开思考、命令或工具详情。
3. 等待新事件和其中一个 Agent 先结束。

验收：

- [ ] 较早详情不会自动折叠。
- [ ] 页面不会无故跳回底部或向下跳一段。
- [ ] 已结束 Agent 的最终 usage 正常显示，另一个 Agent 继续运行不影响它。
- [ ] Stop、Steer、审批和提问只作用于选中的 Agent。

## 5. 重启与恢复

1. 房间空闲时关闭并重新打开开发版。
2. 重新进入同一房间。

验收：

- [ ] 房间仍只显示一个侧栏会话，不出现每个 Agent 的重复会话。
- [ ] Agent 名称、模型、默认接收者和历史分区保持不变。
- [ ] 若某个 Agent 恢复失败，只显示该 Agent 的错误，其他 Agent 仍可使用。

## 6. 搜索、导出、归档与找回

- [ ] 搜索次要 Agent 内容时，结果显示正确 Agent/模型并定位到对应事件。
- [ ] Markdown 导出包含房间 Agent 表、用户接收者和每个回复的 Agent/模型。
- [ ] Kimi 调试包导出先要求选择具体 Agent。
- [ ] 房间归档/恢复显示每个 Agent 的成功或失败结果。
- [ ] 搜索 → “待找回 Agent”可独立打开未绑定或元数据异常的官方 session。

## 7. 回传内容

请至少回传：

- 2 Agent 正常宽度截图。
- 4 Agent + 长名称截图。
- 窄窗口展开 Agent 选择器截图。
- 一次真实并行完成后的完整对话截图。
- 若出现问题：操作步骤、涉及 Agent/模型、是否正在运行、错误文案，以及问题前后的截图。

验收通过前不推送 `master`、不打 tag、不发布 Release。
